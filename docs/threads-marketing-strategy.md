# 정책알리미 쓰레드 마케팅 전략 (v0.1)

## 이 문서가 뭔가요

정책알리미(keepioo.com)와 BlogFury(blog-automation-ten.vercel.app) 두 SaaS를 쓰레드(Threads)로 홍보해서 유료 결제·트래픽을 만드는 전략 합의서. 사업계획서(`business-plan.md`)의 **"M3 SEO 콘텐츠 + M4 커뮤니티 마케팅"** 단계의 한 축으로, 쓰레드 채널을 어떻게 운영할지를 구체화한다.

**목표:** 6개월 안에 정책알리미 유료 구독자 50명 도달에 기여. 쓰레드를 무료 가입 트래픽의 핵심 채널 중 하나로 만든다.

---

## 1. 큰 방향 (합의된 것)

| 항목 | 결정 |
|---|---|
| 메인 제품 | **정책알리미** (BlogFury는 부수 노출) |
| 첫 타겟 세그먼트 | **소상공인·자영업자** (정책자금·대출·임대료·카드수수료 등) |
| 콘텐츠 운영 모델 | **모델 2 — 자동화 파이프라인** (정책알리미 DB → AI → 쓰레드 자동 발행) |
| 톤·서사 | **빌드 인 퍼블릭** (운영자 서사. "혼자 SaaS 키우는 사장님" 페르소나) |
| 솔직도 | **미디엄 오픈** — 가입자 수·알림 발송 수·정책 수는 공개, **금액(매출·광고비)은 비공개** |
| 발행 빈도 | **주 4-5회** (4-6주 검증 후 데일리로 확장) |
| 자동도 | **글 5편 중 4편 자동 발행 / 1편(사장님 일기)만 수동 다듬기** — 사장님 매주 작업 시간 약 30분 |

**왜 소상공인인가:** ① 결제 의사 가장 높음 ② AI가 만들기 좋은 사실·숫자·조건 중심 콘텐츠와 궁합 ③ 육아·청년 대비 가짜 페르소나 위험 적음.

**왜 빌드 인 퍼블릭인가:** ① 사용자가 처음 요청에서 "스토리"를 명시 ② 쓰레드 알고리즘이 "정보 글"보다 "사람 이야기"를 강하게 밀어줌 ③ 결제 전환의 진짜 동력은 "이 사람이 만든 거라 한 번 써본다".

## 2. 콘텐츠 시리즈 구성 (주간 캘린더)

| 요일 | 시리즈 | 자동도 | 형식 |
|---|---|---|---|
| **월** | 지난주 운영 리포트 | 100% 자동 | "지난주 무료 가입 N명, 알림 N건 발송, 새 정책 N개 추가" + 한 줄 인사이트 |
| **화** | 소상공인 정책 1건 깊이 | 100% 자동 | DB에서 1건 뽑아 — 누가/얼마/어떻게 신청 + 마감일 |
| **목** | 소상공인 정책 1건 깊이 | 100% 자동 | 화요일과 동일 포맷, 다른 정책 |
| **금** | 사장님 운영 일기 | AI 초안 → 사장님 5분 다듬기 | 만들면서 겪은 일·고민·발견. 빌드 인 퍼블릭의 핵심 글 |
| **토 (선택)** | 이번 주 마감 임박 5개 | 100% 자동 | 다음 주 마감 정책 큐레이션 |

**총 4-5개/주. 사장님 작업 시간 주 30분 (금요일 일기 다듬기만).**

## 3. 콘텐츠 톤·말투 가이드

- **1인칭 "저"** — "정책알리미 만든 사장님"으로서 글 씀
- **숫자를 적극 쓰되 금액은 빼기** — "지난주 가입자 17명" O, "지난주 매출 12만원" X
- **결론·핵심을 첫 줄에** — 쓰레드는 첫 줄로 스크롤 멈춤이 결정됨
- **감정 강요 금지** — "정말 감동적이지 않나요?" 류 X. 사실·숫자·관찰만으로 강함을 만듦
- **AI 티 안 나게**: 무난한 정리·요약체 X, 한 가지 발견·한 가지 의견·한 가지 숫자에 집중

## 4. CTA·유입 경로

| 글 종류 | CTA 방식 | 이유 |
|---|---|---|
| 평일 정책 정보 (화·목) | **본문에 링크 없음** + 끝에 "프로필 링크에서 알림 받기" 한 줄 | 쓰레드 외부 링크 페널티 회피, 매 글이 광고처럼 안 보임 |
| 운영 리포트 (월) | 링크 없음. "이번 주도 운영해봅니다" 톤 | 자연 노출 |
| 사장님 일기 (금) | 월 1-2회 **리드 마그넷** 푸시 — 무료 PDF (예: "소상공인이 놓치기 쉬운 정책 50선") → 가입해야 받음 | 무료 → 유료 깔때기의 진입점 |
| 토 마감 임박 글 (선택) | 첫 댓글에 링크 | 본문 알고리즘 보호 |

**별도 채널 — BlogFury 일타쌍피:**
- 정책 정보 글을 BlogFury로 **네이버 블로그 자동 발행**
- 쓰레드에서 "이거 BlogFury로 만들었어요" 식 자연 노출 → BlogFury도 동시 노출
- 두 프로젝트가 **서로 마케팅해주는 구조**

## 5. 첫 1주차 게시글 (스토리 시작점)

**Day 1 (월) — 도전 선언:**
> "오늘부터 정책알리미를 키워봅니다. 6개월 안에 유료 구독자 50명을 목표로요. 시작 시점: 무료 가입 0, 유료 0, DB에 등록된 정책 [N]건. 매주 월요일에 운영 리포트 올릴게요. 1인 개발자가 정부 정책 알림 SaaS 키우는 과정, 같이 보실래요?"

**Day 2 (화) — 첫 정책 정보 글:**
DB에서 가장 반응 좋을 만한 소상공인 정책 1개 픽 → 깊이 풀이.

**Day 3 (목) — 두 번째 정책 정보 글:**
다른 카테고리 정책 1개 (예: 임대료 지원).

**Day 4 (금) — 첫 사장님 일기:**
> "쓰레드 시작한 지 3일째. 솔직히 무서웠어요 — 가입자 0명에서 시작 공개하는 거. 그런데 [관찰 한 줄]. 다음 주에 [구체적 작업] 해볼게요."

**Day 5 (토) — 마감 임박 큐레이션:**
다음 주 마감 정책 5개 묶음.

## 6. 자동화 파이프라인 (시스템 설계 요약)

상세 구현은 별도 implementation plan에서 다룸. 여기는 큰 그림만:

```
[정책알리미 DB] 
   ↓ (매일 새 정책 / 마감 정책 / 운영 통계 추출)
[콘텐츠 생성기 (LLM)]
   ↓ (시리즈별 템플릿 적용 → 쓰레드 글 형태로 변환)
[발행 큐]
   ↓ (요일별 시리즈 매칭)
   ├─ 자동 발행 (월·화·목·토): Threads API 또는 예약 발행
   └─ 사장님 승인 대기 (금): 텔레그램/메일로 초안 전송 → 사장님 다듬어서 직접 발행
```

**핵심 의존성:**
- 정책알리미 DB 접근 (이미 있음)
- LLM API (이미 OpenAI 키 보유)
- Threads 게시 자동화 (Threads API or 예약 발행 도구)
- 발행 결과·반응 수집 (도달·좋아요·저장 → KPI 측정)

**보조 지표:**

| 쓰레드 팔로워 | Threads 인사이트 | 200명 | 1,000명 |

| 무료 → 유료 전환 (전체) | 정책알리미 DB | 1명 | 10명 |

## 8. 위험·제약사항

1. **AI가 생성한 정책 정보의 정확도** — 잘못된 정책 정보 자동 발행 시 신뢰 무너짐. **대책:** 정책 정보는 정책알리미 DB의 검증된 필드만 사용 (LLM이 자유 작문 X, 템플릿에 끼워넣기만)
2. **사장님 일기의 진정성** — AI 초안이 가짜처럼 들리면 빌드 인 퍼블릭 신뢰 소멸. **대책:** 매주 사장님이 5분 직접 다듬기 (이 30%가 진정성의 핵심)
3. **Threads API 안정성** — 공식 API가 제한적. 예약 발행 도구(Buffer 등) 백업 옵션 필요
4. **결제 인프라 미완성** — 사업계획서 M2 단계의 토스페이/카카오비즈 승인 안 끝나면 유료 전환 불가. **마케팅 시작 시점은 결제 인프라 완성 후가 이상적**
5. **콘텐츠 피로** — 4-6주 후 같은 패턴이 지루해질 수 있음. 8주차에 시리즈 구성 재검토
