# 어드민 자동화 마스터 로드맵

작성일: 2026-05-07
연관: `2026-05-07-external-auto-publish-design.md` (외부 자동 글쓰기 24h 가동) — 본 로드맵의 사전 작업.

## 1. 배경

사장님 요청: **"어드민 페이지 기능들 전부 다 자동화로 돌아가게 해줘."**

대화 클라리피케이션을 통해 사장님 의도가 다음 4가지 동시 만족이라는 것을 확인.

- **(가)** 어드민 매일/매주 들여다보고 클릭하는 부담 ↓
- **(나)** LLM 자동 판단 + cron 자동화 영역 확장
- **(다)** 어드민 페이지 자체를 SMS/이메일 다이제스트 한 통으로 대체
- **(라)** 사장님이 어드민을 거의 안 봐도 알아서 돌아가는 상태

이 단일 spec 하나로는 너무 큰 작업이라 **6개 sub-project 로 decompose**. 각 sub-project 는 별도 spec → plan → 구현 사이클을 가진다. 본 문서는 이를 묶는 **마스터 로드맵**.

### 1.1 현재 어드민 자동화 상태 (snapshot)

| 페이지 | 현재 자동화 수준 | 주요 사장님 수동 작업 |
|---|---|---|
| `/admin/health` | 자동 cron + SMS alert | 조회 (수동 0) |
| `/admin/wordpress` | 자동 발행 + 검증 트리거 | 사고 시 검토 |
| `/admin/naver-blog` | 큐 자동 enqueue + SMS alert + prompt 복사 | **수동 발행 (네이버 API 부재 — 자동화 불가)** |
| `/admin/news` | LLM 자동 모더레이션 | **사장님 검토 (모호한 카테고리)** |
| `/admin/dedupe` | 0.95+ 자동 confirm | **사장님 검수 (0.95 미만)** |
| `/admin/cron-trigger` | — | **사장님 디버깅용 수동 호출** |

| `/admin/audit` | 조회용 | 검색 |
| `/admin/recommendation-trace` | 진단 도구 | 사고 시 추적 |

## 2. 목표

1. **어드민 진입 빈도 ↓ 80%** — 사장님이 매일 어드민 들어갈 일 자체를 SMS 다이제스트 한 통으로 대체.
2. **자동 판단 영역 확장** — LLM 자동 confirm 임계 낮추되 안전망과 함께 도입.
3. **운영 사고 0** — 자동화 강화 후 사용자 노출 사고 발생 시 즉시 복구 가능한 안전망 구비.

### 비목표 (Out of Scope)

- 네이버 블로그 무인 자동화 — 캡차·약관 위반 (이미 다른 spec 에서 차선책 적용).
- `/admin/cron-trigger` 의 디버깅용 manual 트리거 — 사람 판단 필요.
- 어드민 UI 자체 폐지 — 사장님이 가끔 들어가야 함 (사고·검수). 다만 진입 빈도만 ↓.
- 사용자 노출 위험을 내포한 임계치 무차별 인하.

## 3. 6개 sub-project 정의

### 3.1 #1 어드민 인벤토리 + 클릭 패턴 분석

**목표**: 현재 어드민의 모든 페이지·기능·자동화 수준을 코드 기반으로 inventory 하고, 사장님이 매일 들여다보는 페이지 빈도·클릭 우선순위를 데이터로 식별.

**작업**:
- 어드민 페이지 inventory (코드 기반): 각 페이지의 server action·form·자동/수동 분류표.
- (선택) Vercel logs 또는 GA4 에서 사장님 어드민 페이지 7일 access 빈도 집계.
- 우선순위 표 산출: "사장님 부담 ↑ × 자동화 가능성 ↑" 상위 5건.

**산출물**: `docs/admin-inventory-2026-05.md` (코드 변경 0).

**위험**: 0 (분석만).

**검증**: inventory 표가 13~15 페이지 모두 커버 + 자동화 가능성 색깔 표기 (자동/부분/불가).

**소요**: 30~60분.

---

### 3.2 #2 매일 SMS/이메일 다이제스트 강화

**목표**: 사장님이 매일 어드민에 들어가야만 알 수 있던 정보를 매일 KST 08:00 SMS 한 통 + 매주 월요일 이메일 한 통으로 통합.

**현재 상태**: `/api/cron/daily-digest` 와 `/api/cron/weekly-digest` 가 이미 존재 (`vercel.json`). 일부 KPI 만 발송 중.

**작업**:
- `lib/notifications/daily-digest.ts` 의 KPI 6 → 12 확장:
  - 어제 신규 가입·활성·신규 정책·자동 처리 (이미 있음)
  - 추가: 워드프레스 발행 24h, 네이버 색인 24h, 검토 필요 큐 (news pending·dedupe pending·naver-blog pending), cron 실패 24h, sentry 신규 에러
- 메시지 형식: 90자 SMS + 클릭 가능한 진입 link (사장님 OK 면 그냥 SMS 만 보고 어드민 안 들어감)
- 매주 월 KST 09:00 weekly-digest 이메일 — 7일 trend chart (HTML) 추가.
- 사장님 검토 큐 ≥ 1 일 때만 어드민 진입 link 제시 (불필요 클릭 ↓).

**산출물**: `lib/notifications/daily-digest.ts` 확장, `lib/email/weekly-digest.tsx` 신규.

**위험**: 低 — 알림 채널만 강화, 핵심 운영 영향 0.

**검증**: 사장님 휴대폰에 SMS 도착 + 본문에 12 KPI + 검토 필요 N건 표시.

**소요**: 1~2시간.

---

### 3.3 #3 LLM 자동 판단 임계 낮추기 (위험 中-高)

**목표**: dedupe·news·welfare 등 LLM 판단 영역의 자동 confirm 임계를 낮춰 사장님 검토 큐 ↓.

**현재 임계**:
- dedupe: score ≥ 0.95 자동 confirm (메모리 reference)
- news: LLM 자동 모더레이션 후 사장님 검토 (모호 카테고리만)

**작업**:
- dedupe 임계 0.95 → 0.85: 두 번째 LLM 검증 단계 추가. 0.85+ 1차 통과 → GPT-4o 또는 Claude 의 추가 판단 → 둘 다 confirm 하면 자동 처리.
- news 모더레이션 강화: 카테고리 불명확 50% 이상 자동 분류 가능하도록 LLM prompt 개선.
- welfare 자동 검증 추가: 신규 정책 등록 시 LLM 이 카테고리·자격 자동 매칭 — 사장님 검토 부담 ↓.

**산출물**: 각 LLM 호출 코드 + 임계 config 변경.

**위험**: **中-高**. 임계 잘못 낮추면 오판단이 사용자 노출. 사용자 사고 위험.

**선결 조건**: **#6 (안전망) 먼저 구현 필수.** 안전망 없이 #3 진행 절대 금지.

**검증**: 변경 후 1주 모니터링. 자동 confirm 100건당 사장님 사후 검토에서 오판단 ≤ 1건이면 통과.

**소요**: 2~3시간 + 1주 모니터링.

---

### 3.4 #4 자동 결과 통합 알림 채널

**목표**: 워드프레스·네이버 색인·뉴스·dedupe 자동 처리 결과 + 사장님 확인 필요 항목을 **한 곳**에서 사장님이 한 번에 인지.

**현재 상태**: 각 cron 별 SMS·이메일 분산. naver-queue-alert (KST 09:10), health-alert (KST 09:05), daily-digest (KST 08:00) 등이 따로 발송.

**작업**:
- `daily-digest` 를 단일 진입점으로 통합. naver-queue-alert·health-alert 는 임계 위반 시에만 별도 발송.
- 다이제스트 형식: "오늘 사장님이 봐야 할 것 N건 + 어제 자동 처리 N건". 진입 link 1개 (`/admin` 메인 대시보드).
- `/admin` 대시보드 자체에 "오늘 사장님 처리 필요 N건" 섹션 (이미 일부 있음 — `lib/admin/dashboard-alerts.ts`) 강화.

**산출물**: `lib/notifications/daily-digest.ts` 의 통합 메시지 + `/admin` 메인 dashboard 개선.

**위험**: 低.

**검증**: 사장님이 SMS 한 통 받고 어드민 들어가서 처리 필요 N건이 일치 + 한 페이지에서 다 처리 완결.

**소요**: 1~2시간.

---

### 3.5 #5 수동 cron-trigger 자동화

**목표**: `/admin/cron-trigger` 의 사장님 손이 가는 일부 호출을 자동 cron 으로 전환.

**작업**:
- inventory (#1) 결과로 자주 호출되는 cron-trigger 식별.
- 자동화 가능한 것 (예: 디버깅용 아닌 운영용) → vercel.json cron 추가.
- 디버깅 전용 (수동 검증 필요) → 그대로 유지 + 명확한 라벨 추가.

**산출물**: `vercel.json` cron 추가, `/admin/cron-trigger` UI 정리 (디버깅 전용 라벨).

**위험**: 低.

**검증**: 자동 cron 추가 후 1주 vercel logs 에서 정상 실행률 100%.

**소요**: 1시간.

---

### 3.6 #6 안전망 — 자동 confirm 미검토 항목 추적

**목표**: #3 의 임계 낮춤 부작용 차단. 자동 confirm 된 항목 중 N일간 사장님이 한 번도 사후 검토 안 한 항목을 별도 큐로 노출 → 무작위 샘플 검증.

**작업**:
- `auto_confirm_audit` 테이블 신규 (또는 기존 `audit_logs` 활용): 자동 confirm 시점·검토자·검토 시점 기록.
- 매주 cron `/api/cron/auto-confirm-audit-sample` — 7일 미검토 자동 confirm 항목 중 무작위 5건 샘플링 → 사장님 검토 큐.
- 사장님 검토 결과 (정상/오판단) 기록 → 임계 자동 조정 신호.
- 오판단 비율 임계 초과 (예: 5%) 시 자동 임계 ↑ rollback.

**산출물**: 마이그레이션 1건, cron 1건, 어드민 검토 페이지 1건.

**위험**: 0 (안전망이라 자체적으로 위험 없음).

**선결**: 없음. **#3 보다 먼저 구현 필수.**

**검증**: #3 적용 후 무작위 샘플 5건 사장님 검토 결과에서 오판단 ≤ 0.

**소요**: 1~2시간 + DDL 1건.

## 6. 진행 방식

각 sub-project 는 별도 spec 작성·구현·검증 사이클:

```
sub-project N 시작
  ├─ 별도 spec 작성 (docs/superpowers/specs/2026-MM-DD-admin-automation-N-{name}-design.md)
  ├─ commit (docs only)
  ├─ 구현 commit 1~2건
  ├─ 검증
  └─ 메모리 update
```

본 마스터 로드맵은 진행 중 sub-project 완료 시마다 § 7 "진행 현황" 섹션을 update 하며 사장님께 다음 sub-project 시작 시 승인 받음.

## 8. 위험·운영 메모

- **사용자 노출 위험**: #3 만 中-高. 다른 sub-project 는 모두 低 또는 0.
- **사장님 핸드 오프 시점**: 임계치 낮추기 (#3) 진행 시 사장님 1주 모니터링 부담.
- **롤백 전략**: 각 sub-project commit 단위로 즉시 revert 가능. 임계 변경 (#3) 은 환경변수 또는 config 로 즉시 조정.
- **운영 비용**: SMS 다이제스트 약 22~30원/건 × 30 = 월 660~900원. weekly-digest 이메일 무료.

## 9. 결정·트레이드오프 메모

- **"전부 자동화"의 진짜 의미** = 사장님 진입 빈도 ↓ 80% + 자동 판단 영역 ↑ + 안전망 강화. 100% 자동은 환상이며 사장님 가드를 없애면 사용자 사고 위험.
- **마스터 spec 1개 + 6개 sub-spec 분리 모델** 선택 이유: brainstorming 스킬의 decompose 가이드라인 + keepioo 의 phase 별 commit 패턴 일치.
- **#3 우선순위가 가장 낮은 이유**: 가장 효과 크지만 가장 위험. 안전망 없이 시작하면 사용자 노출 사고 즉시 발생 가능.
